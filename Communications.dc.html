Communications

Les communications expliquent le projet sans le réduire: notes de lancement, changelog, messages professionnels et résumés reliés aux preuves.

audienceVisiteurs, contacts professionnels, collaborateurs potentiels, lecteurs techniques.
tonClair, sérieux, personnel sans devenir privé, concret sans surpromettre.
preuveChaque annonce devrait pointer vers une page, un gate, une capture ou une décision.
00

Rôle

Un projet important a besoin d'un langage public. Cette page définit ce que les communications doivent contenir.

launch

Note de lancement

Présente le projet, son périmètre courant, les surfaces à inspecter et les limites assumées.

change

Changelog

Explique ce qui change en termes de contenu, comportement, architecture et preuve.

profile

Résumé professionnel

Formule le projet comme preuve de démarche: design, architecture, documentation, vérification.

01

Cadre de message

Chaque communication devrait pouvoir être courte, utile et traçable.

Message public
  = what changed
  + why it matters
  + what can be inspected
  + what remains non-canon or unfinished
  + link to evidence
02

Messages prêts à durcir

Ces formats pourront devenir de vraies entrées publiques après la prochaine passe de contenu.

FormatContenuPreuve attendue

03

Note de lancement prête

Un exemple concret, déjà conforme au cadre de message et au profil de langue par défaut.

gnu.in-OS — public documentation & showcase

What it is: a shell research corpus where surfaces, engine data, motion,
assets, and visual proof advance inside one legible system.
Inspect now: Index (map), Central (live simulation), Atlas (static canvases),
Context (reactive guides), Docs (project, method, assets, evidence, roadmap).
Canon: cube favicon, shared navigation, current mascot, bilingual EN/FR profile.
Not promised: broad install, candidate mascot promotion, unproven claims.
Evidence: local smoke gates + captures + port-data fixtures.
Read next: Project -> Central Live -> Evidence.
04

Ton public

Le style peut rester personnel, mais le lecteur doit sentir une maîtrise et une responsabilité.

faire

Dire ce qui est inspectable

Nommer les pages, les assets, les captures, les gates et les limites visibles.

éviter

Écrire depuis le fil privé

Éviter les formulations qui supposent que le lecteur connaît les sessions internes.
